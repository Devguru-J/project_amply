# Amply — Decisions

> "왜 이렇게 만들었는가." 의사결정 기록 (가벼운 ADR).
> 새 결정은 위에 추가, 번호는 시간 역순.

## D-006 · 서비스명: Amply

**날짜:** 2026-05-04
**상태:** 채택

초기 코드네임은 `Clubtable`이었지만 GitHub 리포 (`project_amply`) 와 정합 + 짧고 발음 쉬움 + "Amplify" 어원으로 **Amply** 채택. 슬로건 ("같이 듣는 순간, 방이 된다.") 은 유지.

**대안 비교:**
- `Clubtable` — 의미 명확하지만 길고 GitHub 명과 다름.
- `Roundtable` — DJ 로테이션 컨셉 살리지만 회의 도구 연상.
- `Amply` — 짧고 음향 도메인 환기 + 리포명과 1:1.

## D-005 · 모든 쓰기는 SECURITY DEFINER RPC로

**날짜:** 2026-05-04
**상태:** 채택

`tracks` / `dj_queue` / `track_reactions` 테이블은 RLS로 직접 INSERT/UPDATE/DELETE를 차단하고, 비즈니스 로직이 들어간 RPC만 허용한다.

**근거:**
- "차례인 사람만 트랙을 추가할 수 있다" 같은 검사를 RLS policy로 표현하면 복잡하다.
- 큐 로테이션처럼 여러 행을 atomic하게 바꿔야 하는 연산은 함수가 자연스럽다.
- 클라이언트 SDK가 우회할 수 없도록 강제됨.

**비용:**
- 새 액션 추가할 때마다 RPC 정의 필요.
- RPC는 single-shot이라 트랜잭션을 클라이언트 흐름과 묶기 어려움. (MVP에선 문제 없음.)

## D-004 · 시간 동기화 X

**날짜:** 2026-05-04
**상태:** 채택 (MVP 한정)

여러 클라이언트의 재생 위치를 ms 단위로 맞추지 않는다. 모두 트랙 시작 시점부터 자체적으로 재생.

**근거:**
- 정확한 동기화는 어렵고 (브라우저 throttling, autoplay policy, 버퍼링 차이) MVP에서 구현하면 다른 모든 일정이 밀린다.
- 사용자 가설: "같은 곡을 같은 방에서 듣는다"는 사회적 감각이면 충분하다.
- Plug.dj도 초창기에는 비슷했다.

**바꿀 트리거:**
- 사용자 피드백에서 "5초 차이가 거슬린다"는 응답이 N회 이상.

## D-003 · 재생 마스터 = 현재 DJ 한 명

**날짜:** 2026-05-04
**상태:** 채택

곡 종료를 보고 `advance_queue` RPC를 호출하는 것은 현재 DJ 클라이언트만 한다.

**근거:**
- 모든 클라이언트가 호출하면 N번 advance 되어 큐가 흐트러짐.
- "방장"이라는 영구 권한자보다, "지금 DJ"가 자연스럽고 분산적이다.
- 현재 DJ 브라우저가 닫혀도 누구든 **스킵** 누르면 advance 되므로 데드락 없음.

**대안:**
- 서버 cron으로 트랙 종료 시점에 advance — 인프라 추가 필요. 기각.
- 모든 클라이언트가 호출하되 DB 단에서 멱등 보장 — `for update` 락은 있지만 race window 존재. 채택 안 함.

## D-002 · DJ 로테이션은 "C안 하이브리드"

**날짜:** 2026-05-04
**상태:** 채택

DJ 큐에 줄을 서고, 차례인 사람이 한 곡을 즉시 추가 → 종료 시 큐 head 로테이션.

**대안:**
- (A) Turntable.fm 풀 모델 — 각 DJ가 자기 트랙 라이브러리에서 한 곡씩. 데이터 모델 + UX 복잡.
- (B) Jukebox — 하나의 트랙 큐, DJ는 추가 권한자. 사회적 의식이 약함.
- (C) **하이브리드 — 채택.** 줄서기 의식 + 단순 데이터 모델.

## D-001 · 별도 백엔드 없이 Supabase 단일 의존

**날짜:** 2026-05-04
**상태:** 채택

Socket.io / 별도 Node 서버를 두지 않고 Supabase Realtime + Postgres + Auth만 사용.

**근거:**
- MVP 운영 부담 최소화.
- Realtime이 Presence / Broadcast / Postgres CDC를 다 커버.
- Vercel + Supabase 조합으로 무료~매우 저렴하게 시작.

**바꿀 트리거:**
- 한 방 동시접속 100+ 가 일상이 되면 Realtime 비용/한계 검토.
- 서버사이드에서 BPM 추정·외부 API 통합이 필요해지면 Edge Functions 또는 별도 워커.
