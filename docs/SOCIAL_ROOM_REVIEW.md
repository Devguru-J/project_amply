# Amply — Social Room Review

작성일: 2026-05-05

## 요약

Amply는 이미 Turntable.fm / Plug.dj의 핵심 골격을 꽤 잘 잡고 있다. 현재 구현은 **방 생성, 게스트 입장, YouTube 재생, DJ 큐, 채팅, 반응, Presence 접속자 표시**까지 갖춘 MVP다. 즉 “음악을 한 명이 틀고, 사람들이 방에서 같이 반응한다”는 기본 루프는 살아 있다.

다만 지금 화면은 아직 “음악방 앱”에 더 가깝고, 예전 turntable.fm이나 plug.dj처럼 **사람들이 같은 장소에 서 있고, 누가 DJ 부스에 있고, 관객이 리액션하는 현장감**은 약하다. 이 부분은 불가능하지 않다. 오히려 현재 Supabase Presence 구조 위에 얹기 좋은 기능이다.

가장 추천하는 다음 방향은 대규모 기능 추가보다, 룸 화면을 **공간화**하는 것이다.

1. 접속자 패널을 단순 목록에서 “룸 플로어”로 확장한다.
2. DJ 큐를 사람들의 줄 또는 부스 근처 대기열처럼 보이게 한다.
3. 좋아요/싫어요를 숫자 카운터가 아니라 방 안에서 터지는 리액션으로 만든다.
4. 현재곡 진행, 다음 DJ 차례, 채팅 분위기가 한 장면 안에서 느껴지게 한다.

## 사이트 전체 진단

### 1. 첫인상은 좋지만 “지금 들어갈 곳”이 약하다

랜딩의 카피와 톤은 좋다. “같이 듣는 순간, 방이 된다”는 문장은 제품을 잘 설명하고, 다크 클럽 톤도 방향이 분명하다. 다만 첫 방문자가 가장 먼저 궁금해할 질문은 이것이다.

> 지금 실제로 누가 듣고 있나?

현재 랜딩은 멋진 소개 화면에 가깝고, 라이브 서비스의 열기는 아직 충분히 드러나지 않는다. 공개방이 비어 있으면 더 그렇다. 음악 소셜 서비스는 빈 상태가 치명적이다. 사용자는 “멋진 앱이네”보다 “어? 지금 들어가볼까?”를 느껴야 한다.

개선 제안:

- 히어로 영역 안에 실제 공개방 1개를 더 크게 노출한다.
- 방 카드에 현재곡, 현재 DJ, 접속자 수, 큐 인원, 최근 반응 수를 보여준다.
- 공개방이 없을 때도 “데모 라운지” 또는 “오늘의 라운지”를 자동으로 하나 보여준다.
- `NOW LIVE` 문구는 실제 live 데이터와 연결한다. 방이 없으면 다른 카피를 쓴다.
- 랜딩 CTA는 “공개방 둘러보기”보다 “지금 듣고 있는 방 입장”이 더 강하다.

추천 우선순위: 높음. 첫인상은 디자인보다 **살아있다는 증거**가 더 중요하다.

### 2. 온보딩은 짧지만, 입장 후 다음 행동 안내가 필요하다

게스트 닉네임 입장은 아주 좋다. 이 제품에는 회원가입 없는 진입이 맞다. 다만 방에 들어간 뒤 사용자가 해야 할 행동이 조금 더 명확해야 한다.

현재 사용자는 입장 후 이런 질문을 할 수 있다.

- 음악이 안 나오면 내가 뭘 해야 하지?
- DJ 참여는 무슨 뜻이지?
- 곡은 언제 추가할 수 있지?
- YouTube URL은 어디서 넣지?
- 내가 지금 듣는 사람인지, DJ 차례인지 어떻게 알지?

개선 제안:

- 첫 입장 시 방 상태별 빈 화면을 다르게 보여준다.
- 큐가 비어 있으면 “첫 DJ로 부스에 서기”를 중심 CTA로 둔다.
- 내가 큐 1번이고 곡이 없으면 “지금 한 곡 틀기”를 화면 중앙에 띄운다.
- 내가 큐에 있지만 차례가 아니면 “2번째로 대기 중”처럼 차례를 명확히 보여준다.
- 첫 메시지/첫 반응/첫 DJ 참여를 작은 체크리스트처럼 유도한다.

추천 우선순위: 높음. 소셜 앱의 MVP는 기능이 있어도 사용자가 첫 30초 안에 루프를 이해하지 못하면 이탈한다.

### 3. 공개방 목록은 정보 밀도가 부족하다

현재 공개방 카드는 이름과 설명, LIVE 표시 중심이다. 하지만 음악방 목록에서 중요한 건 방 이름보다 **지금 어떤 음악과 사람들이 있는지**다.

개선 제안:

- 현재곡 제목과 썸네일
- 현재 DJ 이름
- 접속자 수
- DJ 큐 길이
- 최근 5분 반응 수
- 방 테마 또는 장르 태그
- 마지막 활동 시간

공개방 정렬도 단순 생성일 기준보다 “현재 살아있는 방” 중심이 좋다.

추천 정렬:

1. 현재 트랙이 재생 중인 방
2. Presence 접속자가 있는 방
3. 최근 채팅/반응이 있는 방
4. 최근 생성된 방

이걸 위해서는 장기적으로 `rooms.last_active_at`, `rooms.listener_count_cache`, `rooms.current_track_id` 기반 목록 최적화가 필요할 수 있다. MVP에서는 클라이언트/서버에서 현재 DB로 가능한 만큼만 보여줘도 된다.

### 4. 방 만들기는 간단하지만 “방의 성격”을 만들 수 없다

현재 방 만들기는 이름, 설명, 공개 여부만 받는다. MVP로는 충분하지만, 음악방은 “무슨 분위기의 방인지”가 입장 결정에 중요하다.

개선 제안:

- 장르/무드 태그: 새벽, 인디, 전자음악, 힙합, 작업용, 신청곡
- 방 테마: 어두운 클럽, 라운지, 라디오, 하우스 파티
- 운영 방식: 자유 DJ / 호스트 승인 / 친구 전용
- 기본 스킵 정책: 호스트만 / 투표 / DJ만

처음부터 복잡한 설정 화면으로 만들 필요는 없다. “이 방의 무드” 한 줄 선택만 있어도 충분하다.

추천 우선순위: 중간. 공개방이 많아지기 전에는 급하지 않지만, 방의 개성을 만드는 데 중요하다.

### 5. 모바일 경험은 반드시 실사용 검증이 필요하다

이 제품은 모바일 사용 가능성이 높다. 친구가 링크를 보내면 대부분 휴대폰에서 열어볼 것이다. 현재 룸 화면은 기능이 많아서 모바일에서 다음 문제가 생길 수 있다.

- YouTube, 채팅, 큐, 반응이 세로로 길게 쌓여 핵심 행동이 묻힘
- 채팅 입력 시 키보드가 올라오면서 레이아웃이 밀림
- DJ 큐와 반응 버튼이 스크롤 아래로 내려가 발견성이 낮아짐
- iOS Safari 자동재생 정책 때문에 입장 직후 무음/재생 실패가 발생할 수 있음

개선 제안:

- 모바일 룸은 하단 고정 컨트롤을 둔다: 반응, DJ 참여, 채팅 열기
- 채팅은 전체 화면 bottom sheet로 여는 패턴도 고려한다.
- YouTube 영상보다 NowPlaying + RoomStage를 우선 배치하는 모바일 전용 구조를 검토한다.
- “소리가 안 나면 탭해서 재생” 같은 상태 안내를 명확히 둔다.

추천 우선순위: 매우 높음. 외부 테스트 전에 모바일은 꼭 봐야 한다.

### 6. 랜딩 카피는 좋은데 기능 설명이 조금 추상적이다

현재 카피는 감성은 좋다. 다만 turntable.fm/plug.dj를 모르는 사람에게는 “그래서 내가 정확히 뭘 하나?”가 조금 늦게 온다.

개선 제안:

- “닉네임으로 입장”
- “DJ 줄에 서기”
- “내 차례에 YouTube 링크 한 곡”
- “곡이 끝나면 다음 사람”
- “채팅과 반응으로 같이 듣기”

이 5단계가 랜딩 중간에 짧게 보이면 제품 이해가 빨라진다. 지금 `Features` 섹션이 그 역할을 일부 하지만, 더 구체적인 플로우로 바꿔도 좋다.

### 7. 신뢰/안전 장치가 아직 부족하다

공개방, 채팅, 게스트 입장이 있는 서비스는 작아도 운영 장치가 필요하다.

개선 제안:

- 호스트 권한: 유저 큐 제거, 채팅 메시지 숨김, 방 닫기
- 기본 신고: 메시지 신고, 유저 신고
- 게스트 rate limit: 메시지 연속 전송 제한, 방 생성 제한
- 금칙어/스팸 최소 방어
- 비공개방 초대 링크 만료

초기에는 무거운 신고 시스템보다 “호스트가 방을 지킬 수 있는 도구”가 먼저다.

### 8. 유지/성장 루프가 아직 약하다

음악방은 한 번 들어오고 끝나면 아쉽다. 다시 오게 만드는 루프가 필요하다.

개선 제안:

- 방 세션 recap: 오늘 나온 곡 리스트와 베스트 반응
- DJ 프로필: 내가 튼 곡, 받은 좋아요, 자주 가는 방
- 공유 카드: “어제 새벽 라운지에서 나온 12곡”
- 즐겨찾는 방
- 최근 입장한 방
- 데일리 라운지

이 기능들은 당장 MVP의 필수는 아니지만, 서비스가 “한번 써본 장난감”에서 “다시 여는 장소”가 되게 만든다.

### 9. SEO/공유 최적화는 기본은 있으나 룸별 확장이 필요하다

현재 루트 메타데이터는 설정되어 있다. 다음 단계는 룸별 공유 카드다.

개선 제안:

- `/rooms/[roomId]` 동적 metadata
- 룸 이름/설명 기반 OG title/description
- 현재곡 또는 방 테마 기반 OG 이미지
- 공개방 리스트 sitemap
- 비공개방은 색인 제외

음악방은 링크 공유로 퍼질 가능성이 크기 때문에, 카카오톡/디스코드/트위터에 붙였을 때 매력적인 카드가 중요하다.

### 10. 분석 지표가 필요하다

다음에 무엇을 만들지 결정하려면 최소한의 이벤트 로그가 필요하다.

추천 이벤트:

- 랜딩 방문
- 공개방 클릭
- 게스트 입장 성공
- 방 생성
- DJ 큐 참여
- 곡 추가 성공/실패
- 채팅 전송
- 반응 클릭
- 스킵
- 1분 이상 체류
- 같은 방 재방문

초기에는 복잡한 분석 도구보다 간단한 이벤트 테이블이나 Cloudflare Web Analytics 정도로 시작해도 된다.

## 현재 구현에서 좋은 점

### 1. 제품 원칙이 명확하다

`docs/PRODUCT.md`의 “한 명이 틀고 모두가 듣는다”, “로테이션이 의식이다”, “MVP는 동기화하지 않는다”는 결정이 좋다. 이 서비스는 음원 스트리밍 기술보다 **차례, 반응, 같이 있음**이 더 중요하다.

### 2. 실시간 구조가 확장 가능하다

현재 `src/lib/realtime.ts`에는 Presence와 Postgres Changes가 이미 분리되어 있다.

- Presence: 지금 방에 누가 있는지
- Postgres Changes: 현재 트랙, 큐, 채팅, 반응
- Broadcast: 아직 거의 쓰지 않지만, 향후 즉시성 이벤트에 사용 가능

아바타 위치, 박수, 춤, 이모트, DJ 입장 같은 일시적인 이벤트는 DB에 전부 저장하지 않고 Broadcast / Presence 메타데이터로 처리할 수 있다. 이 점이 중요하다.

### 3. MVP 범위가 과하지 않다

YouTube IFrame Player에 음악을 위탁하고, 앱은 사회적 경험에 집중하는 선택이 맞다. Spotify나 자체 음원 재생으로 바로 가면 라이선스, 계정, 프리미엄 권한, 모바일 자동재생 정책 때문에 일이 급격히 커진다.

### 4. 룸 UI의 구성 요소가 이미 존재한다

현재 룸 화면에는 다음 요소가 있다.

- `YouTubePlayer`: 재생 화면
- `NowPlaying`: 현재곡과 DJ
- `ReactionBar`: 좋아요/싫어요
- `DJQueue`: DJ 차례
- `ChatPanel`: 실시간 채팅
- `ParticipantsPanel`: 접속자 목록

이 구조는 “공간형 룸”으로 바꾸기 좋은 재료다. 새로 갈아엎기보다, 기존 컴포넌트 사이에 **Stage / Floor 레이어**를 추가하는 식이 자연스럽다.

## 아쉬운 점과 개선 방향

### 1. 접속자가 “사람”이 아니라 “목록”으로 보인다

현재 `ParticipantsPanel`은 접속자를 잘 보여주지만, 경험적으로는 Slack의 멤버 목록에 가깝다. Turntable.fm과 plug.dj의 매력은 유저가 작은 아바타로 같은 공간에 있고, DJ가 앞에 서 있고, 사람들이 반응하는 장면이었다.

개선 제안:

- 룸 중앙에 `RoomStage` 또는 `AudienceFloor` 컴포넌트를 추가한다.
- Presence 사용자들을 반원형 또는 클럽 플로어 좌표에 배치한다.
- 현재 DJ는 DJ 부스 위치에 고정한다.
- DJ 큐에 있는 사람은 부스 옆 대기열에 표시한다.
- 일반 청취자는 플로어에 배치한다.
- 내 아바타는 작은 강조 링 또는 “YOU” 배지로 구분한다.

난이도는 생각보다 낮다. 초반에는 3D가 필요 없다. CSS absolute positioning + deterministic layout만으로도 충분히 분위기가 난다.

### 2. 리액션이 숫자로만 남는다

좋아요/싫어요 카운터는 기능적으로 충분하지만, 현장감은 약하다. 음악방에서는 “누가 반응했는지”, “방이 갑자기 달아올랐는지”가 중요하다.

개선 제안:

- 좋아요를 누르면 내 아바타 위에서 짧은 빛/아이콘이 뜬다.
- 여러 명이 좋아요를 누르면 플로어 조명이 밝아진다.
- 싫어요가 일정 이상 쌓이면 스킵 투표 UI가 자연스럽게 등장한다.
- 반응 수를 `ReactionBar`에만 두지 말고, `RoomStage`에도 짧은 애니메이션으로 반영한다.

추천 구현:

- DB에 저장되는 최종 반응은 기존 `track_reactions` 유지
- 순간 애니메이션은 Supabase Broadcast 사용
- 늦게 들어온 사람은 카운터만 보고, 현재 방 안의 사람들은 실시간 애니메이션을 본다

### 3. DJ 차례의 긴장감이 약하다

현재 큐는 잘 동작하지만, “내 차례가 다가온다”는 느낌은 아직 약하다. Turntable.fm류 서비스에서 재미있는 건 내 차례가 가까워질수록 선곡을 고민하게 되는 긴장감이다.

개선 제안:

- 다음 DJ를 `UP NEXT`로 크게 보여준다.
- 내 차례가 가까워지면 30초 전 알림/글로우를 준다.
- 현재 DJ가 곡을 넣지 않으면 타임아웃으로 다음 사람에게 넘긴다.
- 큐를 단순 pill 목록이 아니라 “부스 옆 줄”처럼 보이게 한다.

우선순위는 높다. 이 기능은 소셜 감각뿐 아니라 실제 사용성도 개선한다.

### 4. 현재곡의 공동 재생감이 약하다

문서상 MVP에서 정밀 시간 동기화는 제외되어 있다. 이 결정은 맞지만, 서비스가 커질수록 사용자는 “정말 같이 듣고 있다”고 느끼고 싶어진다.

개선 제안:

- 1차: 현재 `started_at` 기준으로 UI 타임라인만 맞춘다.
- 2차: 새로 들어온 사람은 `started_at`과 현재 시간을 계산해 YouTube를 해당 초로 seek한다.
- 3차: DJ 또는 서버 기준 clock drift 보정을 한다.

주의할 점:

- YouTube 자동재생 정책 때문에 모든 브라우저에서 완벽한 자동 싱크는 어렵다.
- 모바일 Safari, 백그라운드 탭, 사용자의 음소거/자동재생 설정이 변수다.
- 그래도 “대충 같은 구간을 듣는 느낌”까지는 충분히 구현 가능하다.

정밀한 ms 단위 싱크는 어렵지만, 이 앱에 필요한 수준의 느슨한 싱크는 가능하다.

### 5. 스킵 권한이 너무 열려 있다

현재 스키마 기준 `skip_track` RPC는 “누구나” 호출할 수 있다. MVP에서는 괜찮지만 공개방에서는 악용될 수 있다.

개선 제안:

- Phase 4에 적힌 투표 스킵을 먼저 구현한다.
- 현재 DJ 또는 호스트만 즉시 스킵 가능하게 한다.
- 일반 유저는 downvote 또는 skip vote를 누르고, 방 인원의 일정 비율 이상이면 자동 스킵한다.

추천 기준:

- 2명 이하 방: 호스트/DJ 스킵만 허용
- 3명 이상 방: 현재 접속자 중 50% 이상 또는 3표 이상일 때 스킵
- 스킵 투표는 트랙마다 1인 1표

### 6. 방에 “기억”이 부족하다

음악방은 끝나고 나면 트랙리스트가 남아야 다시 공유하고 싶어진다. 지금은 `tracks` 테이블에 히스토리가 쌓일 수 있지만 UI에서 경험으로 드러나지 않는다.

개선 제안:

- 최근 재생곡 히스토리 패널
- “오늘 이 방에서 가장 반응 좋았던 곡”
- 방 세션 종료 후 공유 가능한 recap 페이지
- DJ별 선곡 기록
- 좋아요가 많이 나온 순간 타임라인

이 기능은 재방문 이유를 만든다. 단기 MVP 이후에는 꽤 중요하다.

## 아바타 룸 구현은 어렵거나 불가능할까?

불가능하지 않다. 오히려 이 프로젝트에 잘 맞는다. 다만 어떤 수준을 목표로 하느냐에 따라 난이도가 크게 달라진다.

### 옵션 A: 2D 아바타 플로어

추천도: 매우 높음
난이도: 낮음-중간
예상 작업: 작게 시작하면 1-2일

방 안에 2D 좌표계를 만들고 아바타를 배치하는 방식이다.

구현 아이디어:

- `PresenceUser`에 `avatar_seed`, `mood`, `last_reaction`, `floor_slot` 같은 값을 추가한다.
- 별도 DB 테이블 없이 Presence payload만으로 시작한다.
- 화면에는 `RoomStage` 컴포넌트를 만들고, 접속자를 좌표에 배치한다.
- 좌표는 랜덤이 아니라 `user_id` 해시 기반으로 결정해 새로고침해도 같은 위치에 둔다.
- 현재 DJ는 중앙 상단 DJ 부스에 배치하고, 나머지는 아래쪽 플로어에 배치한다.

장점:

- 가장 빠르게 현장감이 난다.
- DB 마이그레이션 없이도 시작 가능하다.
- 모바일 대응이 쉽다.
- 성능 부담이 낮다.

주의점:

- 사람이 많아지면 겹침 처리가 필요하다.
- 20-30명 이상 공개방까지 고려하면 슬롯 시스템이 필요하다.

### 옵션 B: 2D 아바타 + 이모트/댄스 상태

추천도: 높음
난이도: 중간
예상 작업: 2-4일

옵션 A에 “살아있는 반응”을 얹는 방식이다.

구현 아이디어:

- 좋아요 클릭 시 Broadcast로 `{ type: "emote", user_id, kind: "up" }` 전송
- 각 클라이언트는 해당 유저 아바타 위에 800ms짜리 이펙트 표시
- 현재곡의 up/down 비율에 따라 플로어 조명 색 변화
- 채팅 입력 중이면 아바타 옆에 작은 말풍선 점 표시
- DJ가 곡을 시작하면 DJ 아바타에 spotlight 애니메이션

장점:

- turntable.fm/plug.dj식 감성이 가장 빨리 살아난다.
- DB write를 늘리지 않고 Broadcast로 처리 가능하다.
- 사용자가 “내가 방에 영향을 주고 있다”고 느낀다.

주의점:

- Broadcast 이벤트는 영속적이지 않으므로 새로 들어온 사람은 과거 이펙트를 못 본다. 이건 오히려 자연스럽다.
- 너무 많은 애니메이션은 산만해질 수 있으니 절제해야 한다.

### 옵션 C: 커스터마이즈 가능한 아바타

추천도: 중간-높음
난이도: 중간
예상 작업: 3-5일

프로필에 아바타 스타일을 저장하는 방식이다.

구현 아이디어:

- `profiles`에 `avatar_style`, `avatar_seed`, `accent_color` 추가
- DiceBear 같은 생성형 SVG 아바타 사용
- 게스트는 닉네임 기반 deterministic seed로 자동 생성
- 로그인 유저는 색상/표정/아이템 정도만 선택 가능

장점:

- 재방문 동기가 생긴다.
- 익명 유저도 캐릭터성이 생긴다.
- “이 사람은 자주 오는 사람”이라는 감각을 만들기 좋다.

주의점:

- 커스터마이징 화면을 너무 크게 만들면 핵심 음악 경험에서 벗어난다.
- 초반에는 색상/아이콘/테두리 정도로 충분하다.

### 옵션 D: Canvas 기반 룸

추천도: 중간
난이도: 중간-높음
예상 작업: 1-2주

HTML/CSS 대신 Canvas로 플로어를 그리는 방식이다. 사람이 많거나 이펙트가 많아질 때 유리하다.

장점:

- 수십 명 아바타와 파티클을 더 부드럽게 처리할 수 있다.
- 음악 시각화, 조명, 리액션 이펙트를 만들기 좋다.

단점:

- 접근성, 반응형, 클릭/툴팁 구현이 CSS보다 번거롭다.
- 기존 React 컴포넌트 구조와 분리되는 면이 있다.
- 지금 단계에서는 과할 수 있다.

### 옵션 E: 3D 룸

추천도: 낮음, 나중에 실험
난이도: 높음
예상 작업: 최소 2-4주 이상

Three.js로 클럽 공간을 만드는 방식이다.

가능은 하지만, 지금 제품 단계에서는 추천하지 않는다. 3D는 보기에는 강력하지만 모바일 성능, 카메라, 클릭, 텍스트 가독성, 접근성, 개발 시간이 모두 무거워진다.

추천한다면:

- 핵심 UX가 검증된 뒤 이벤트성 테마로 실험
- 3D 전체 룸보다 “배경 무대” 정도로 제한
- 채팅/큐/재생 컨트롤은 여전히 HTML UI로 유지

## 추천 우선순위

### P0: 실사용 검증 전에 꼭 보면 좋은 것

1. 외부 사람 1명과 실제 공개 URL에서 같이 듣기
2. 모바일 룸 페이지 확인
3. YouTube 자동재생/음소거 정책 확인
4. 스킵 버튼 권한 최소 조정 또는 투표 스킵 설계 확정
5. 빈 방/빈 공개방 상태 개선

### P1: 현장감을 만드는 첫 번째 패키지

1. `RoomStage` 2D 아바타 플로어
2. DJ 부스와 현재 DJ spotlight
3. 플로어 접속자 배치
4. 좋아요/싫어요 시 아바타 위 리액션
5. 다음 DJ 강조

이 패키지가 가장 효과 대비 작업량이 좋다. 이걸 넣으면 “음악방 앱”에서 “사람들이 있는 방”으로 인상이 바뀐다.

### P2: 음악방으로 오래 쓰게 만드는 기능

1. 투표 스킵
2. 곡 추가 타임아웃
3. 재생 히스토리
4. 세션 recap 공유 페이지
5. 방 초대 링크
6. 호스트 권한

### P3: 차별화 기능

1. 방 테마
2. DJ 레벨/뱃지
3. 데일리 라운지
4. BPM 또는 에너지 기반 조명
5. 아바타 커스터마이징

## 제안하는 화면 구조

현재 구조:

```text
왼쪽: 방 정보 + 접속자
중앙: YouTube + 현재곡 + 반응
오른쪽: 채팅
하단: DJ 큐
```

추천 구조:

```text
상단: 룸 헤더

중앙 큰 영역:
  RoomStage
    - DJ 부스
    - 현재 DJ 아바타
    - 관객 아바타 플로어
    - 현재곡 조명/리액션 이펙트

오른쪽:
  ChatPanel

하단:
  DJQueue
  ReactionBar

접속자 목록:
  보조 패널로 유지하거나, RoomStage 안의 hover/detail로 흡수
```

YouTube 영상은 반드시 중앙 최상단일 필요가 없다. 이 서비스의 주인공이 “뮤직비디오 보기”가 아니라 “같이 듣기”라면, 영상은 DJ 부스 뒤 스크린처럼 들어가는 편이 더 잘 맞을 수 있다.

## 데이터 모델 추가 후보

초기에는 Presence만으로 충분하다. 그래도 이후를 생각하면 다음 필드를 고려할 수 있다.

### profiles

```text
avatar_seed text
avatar_style text
accent_color text
last_seen_at timestamptz
```

### rooms

```text
theme_id text
max_djs int
max_listeners int
skip_policy jsonb
```

### track_events

```text
id uuid
room_id uuid
track_id uuid
user_id uuid
kind text -- up, down, skip_vote, join_stage, leave_stage
created_at timestamptz
```

`track_events`는 처음부터 만들 필요는 없다. 하지만 recap, 타임라인, 인기 순간을 만들고 싶다면 나중에 도움이 된다.

## 구현 메모

### Presence payload 확장

현재:

```ts
{
  user_id,
  display_name,
  avatar_url,
  online_at
}
```

확장 후보:

```ts
{
  user_id,
  display_name,
  avatar_url,
  online_at,
  avatar_seed,
  accent_color,
  floor_slot,
  status,        // listening | dj | queued | away
  last_emote
}
```

Presence는 “지금 이 순간의 상태”에 적합하다. 위치, 이모트, 입력 중 여부처럼 잠깐 보이는 값은 여기에 잘 맞는다.

### Broadcast 이벤트 후보

```ts
type RoomEvent =
  | { type: "emote"; user_id: string; kind: "up" | "down" | "fire"; at: string }
  | { type: "chat-burst"; user_id: string; at: string }
  | { type: "dj-spotlight"; user_id: string; track_id: string; at: string }
  | { type: "skip-vote"; user_id: string; track_id: string; at: string };
```

Broadcast는 “방금 일어난 일”에 적합하다. 저장하지 않아도 되는 애니메이션과 즉시 피드백을 담당하면 된다.

## UX 카피 제안

현재 톤이 좋으니, 기능명도 제품의 정서에 맞추는 편이 좋다.

- Participants → Here Now 또는 Floor
- DJ Queue → On Deck
- Join DJ Queue → 부스에 서기
- Leave Queue → 줄에서 빠지기
- Add Track → 한 곡 틀기
- Reaction Up → 좋다
- Reaction Down → 넘기자
- Waiting → 다음 DJ를 기다리는 중
- Skip Vote → 넘기기 투표

너무 설명적이면 서비스가 도구처럼 보이고, 너무 감성적이면 사용성이 흐려진다. 버튼은 짧고 명확하게, 보조 문구는 분위기 있게 가는 게 좋다.

## 가장 추천하는 다음 작업

첫 번째로는 `RoomStage`를 만드는 것을 추천한다.

범위는 작게 잡는다.

1. 현재 접속자들을 2D 플로어에 배치한다.
2. 현재 DJ만 DJ 부스 위치에 보여준다.
3. DJ 큐에 있는 사람은 작은 대기열로 보여준다.
4. 좋아요/싫어요를 누르면 아바타 위에 1초짜리 리액션이 뜬다.
5. 기존 `ParticipantsPanel`은 일단 유지한다.

이 정도만 해도 예전 turntable.fm / plug.dj의 “같이 현장에 있는 느낌”이 상당히 살아난다. 3D나 복잡한 커스터마이징은 그 다음이다.

## 결론

아바타가 같이 방에 들어와 음악을 즐기는 느낌은 구현 어렵거나 불가능한 영역이 아니다. 현재 Amply의 구조에서는 **2D 아바타 플로어 + Presence + Broadcast** 조합으로 현실적으로 만들 수 있다.

중요한 건 처음부터 거대한 가상공간을 만들지 않는 것이다. 작은 아바타, DJ 부스, 반응 이펙트, 다음 차례의 긴장감만으로도 충분히 “같이 듣고 있다”는 감각이 생긴다.

Amply의 다음 도약은 기능 수를 늘리는 것보다, 지금 있는 사람들을 화면 안에서 진짜 사람처럼 느끼게 만드는 데 있다.
